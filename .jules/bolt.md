## 2024-05-20 - SemanticText Re-renders
**Learning:** `SemanticText` splits the text and uses `part.match` in its render function, without memoization. The same goes for the regex pattern which is re-created every render. This happens inside `NarrativeLog.tsx` which maps over logs. Every re-render of `NarrativeLog` or the entire App forces this regex split over and over.
**Action:** Extract the static regex out of the component scope into `SEMANTIC_KEYWORD_REGEX`. Memoize the `parts` parsing with `React.useMemo` to avoid executing the regex operation repeatedly on identical log text. Wrap both `SemanticText` and `NarrativeLog` in `React.memo` to skip rendering when props don't change.
## 2024-05-20 - NarrativeLog Component Stable Keys
**Learning:** Using the map index `i` as the `key` prop in React lists that prepend or shift (like `logs.slice(-20)`) causes every element to shift indices on insertion, forcing full unmounting, remounting, and re-triggering of enter animations. This introduces massive, invisible performance penalties and janky UI behavior for what should be simple DOM appends.
**Action:** Use an absolute array index (`startIndex + i`) or a unique identifier from the data model (like an ID) to give stable keys across renders. For append-only lists where we slice the end, `const startIndex = Math.max(0, logs.length - windowSize);` combined with `key={startIndex + i}` preserves React node continuity, ensuring performant appends and stable animations.
## 2024-05-20 - React.memo Failing on Global State and Inline Node Props
**Learning:** Standard `React.memo` is entirely ineffective for components like `NarrativeLog` that accept the entire global `state` object, or components like `StatBar`/`SectionHeader` that accept inline React elements for `icon` props (e.g. `icon={<Heart />}`). Because the reference to the global `state` object or the inline element changes on every render of the parent `App` or `DoLStatsSidebar`, `React.memo` constantly fails the shallow equality check, leading to massive unnecessary re-renders.
**Action:** Always provide custom comparator functions to `React.memo` when components accept complex global objects or inline React nodes as props. For global state, the comparator should strictly check the specific nested properties the component actually consumes (e.g., `prev.state.player.knowledge.sexual_awareness === next...`). For inline nodes, compare only the primitive props (`label`, `value`, `color`) and ignore the `icon` node entirely if it's static.

## 2024-05-20 - Global State Re-renders in React.memo
**Learning:** `StatsSidebar` accepted the entire global `state` object and relied on a default `React.memo` implementation. Since the `state` object reference changes on every single tick (e.g., when a background NPC moves or a generic quest updates), `StatsSidebar` was re-rendering constantly, even though the specific UI elements it displayed (player stats, date, weather) hadn't necessarily changed.
**Action:** When using `React.memo` on components that accept a large global state object, always write a custom comparator function that explicitly checks only the specific nested properties the component actually renders. This prevents massive re-render cascades triggered by unrelated state updates.
## 2025-04-21 - [App callback stabilization for React.memo]
**Learning:** When attempting to use `React.memo` on heavily updated child components (like `NarrativePanel`) that accept callback functions (like `handleAction`) from a parent (`App.tsx`) holding the global state, using a custom comparator that ignores the callback leads to severe stale closure bugs. Conversely, comparing the callback breaks memoization because the callback is recreated on every state tick.
**Action:** Refactor the parent's callback using the `useRef` pattern (`stateRef.current = state`) to access the latest state without including it in the `useCallback` dependency array. This keeps the function reference stable, allowing child components to safely rely on standard `React.memo` shallow equality (or a simple custom comparator) without triggering re-renders or stale data issues.
